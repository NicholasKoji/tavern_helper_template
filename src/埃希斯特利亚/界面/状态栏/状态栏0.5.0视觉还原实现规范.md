# 埃希斯特利亚状态栏 0.5.0 视觉还原实现规范

## 文档目的

本文件用于新会话继续开发状态栏 0.5.0 时交接上下文。

当前目标不是重新发散设计，也不是继续沿用现有前端视觉，而是以 `设计稿/0.5.0状态图` 中保存的 13 张状态图为视觉基准，重做状态栏前端的视觉层。现有代码中的数据读取、Mock 预览、交互状态模型可以保留；视觉表现、布局系统、图标方案需要重新校准。

## 核心判断

当前前端成品与 13 张设计稿偏差较大，主要原因是上一轮实现只落实了信息架构和变量展示，没有把设计稿拆解成硬性视觉规格。后续开发必须先对照设计稿，再写 CSS 和组件结构。

新会话开始后，必须先打开以下文件：

- `ROLE_CARD_AI_HANDOFF_GUIDE.md`
- `src/埃希斯特利亚/设定交接规范.md`
- `src/埃希斯特利亚/界面/状态栏/状态栏交互结构文档.md`
- `src/埃希斯特利亚/界面/状态栏/状态栏图标系统文档.md`
- `src/埃希斯特利亚/界面/状态栏/设计稿/0.5.0状态图/设计稿索引.md`
- `src/埃希斯特利亚/界面/状态栏/设计稿/0.5.0状态图/状态图_01.png`

## 参考优先级

### 第一优先级：变量与设定边界

状态栏只能展示 `schema.ts` 中真实存在的 MVU 变量。不得新增伪功能入口、伪状态、伪警报、伪权限。

禁止重新出现：

- `上路许可`
- `异常警报`
- `档案`
- `位置追踪`
- `通行权限`
- `通知`
- `菜单`
- 与 MVU 变量无关的系统功能入口

### 第二优先级：交互结构文档

交互逻辑以 `状态栏交互结构文档.md` 为准，包括：

- 默认收起态
- 骑士手风琴
- 女骑手风琴
- 场景与待办抽屉
- 装备弹窗
- 机动能力抽屉
- 关系背景、外貌记录、身材记录详情
- 骑士完整详情
- 女骑完整详情总览
- 多人物状态
- SAN 状态规则

### 第三优先级：13 张设计稿

13 张设计稿作为视觉母版，而不是完全硬性照抄交互。

因为这些图是生成图，图与图之间存在少量交互逻辑不一致、按钮层级不稳定、局部变量展示不完整等问题。后续实现时应遵守：

- 视觉风格、组件气质、层级比例尽量贴近设计稿。
- 交互跳转、返回逻辑、变量完整性以交互结构文档为准。
- 当设计稿与真实变量冲突时，以真实变量为准。
- 当设计稿里的按钮或模块暗示不存在功能时，不实现该功能。
- 当设计稿局部信息不完整时，可以按文档补齐，但补齐后的视觉必须融入同一套设计语言。

## 主视觉基准

`状态图_01.png` 是默认态的主视觉基准，必须优先还原。

默认态应呈现为正文后衔接的一张移动端状态卡，不是独立 App 首页。视觉结构如下：

1. 深色主面板
   - 深石墨青、雾黑、冷蓝高光。
   - 外框有厚重但克制的圆角边界。
   - 带轻微内阴影、外投影和材质噪声感。
   - 不要做成普通 dashboard 卡片。

2. 顶部标题
   - 左侧为大号盾徽图标。
   - 主标题为 `埃希斯特利亚`。
   - 右侧为收起/展开符号，而不是多余功能按钮。
   - 不显示 `ESTHISTRIA STATUS` 这种工程化英文眉标，除非设计稿中明确存在。

3. 主角与 SAN 区
   - 主角卡为横向大卡。
   - 左侧：主角盾徽、`主角` 标签、姓名、年龄、身份摘要。
   - 右侧：SAN 圆环读数。
   - SAN 必须是圆形仪表/环形进度，不要改成横向进度条。

4. 场景三栏
   - 时间、地点、天气在同一个深色分组卡中。
   - 三栏之间有细竖线分隔。
   - 图标、标签、主值层级清晰。
   - 地点允许换行，但整体保持稳定高度。

5. 主要待办
   - 使用蓝色强调卡。
   - 左侧待办图标，右侧摘要文本，末尾进入详情箭头。
   - 不要做成普通透明卡。

6. 底部人物列表
   - 使用暖白或浅灰白底板，明显区别于上方深色状态区。
   - 人物行是浅色卡片，不应继续留在深色卡片体系里。
   - 骑士与女骑各一行，显示身份、体力或好感摘要、进度条和展开箭头。
   - 底部白色区域是设计稿默认态的重要识别点，必须保留。

### 状态图 02：骑士手风琴展开态

用于还原骑士行展开后的视觉。重点不是逐字照抄，而是保留：

- 摘要信息呈卡片或分组块。

### 状态图 03：女骑手风琴展开态

用于还原女骑行展开后的视觉。必须包含：

- `查看完整女骑详情` 或同等总览入口。

如果设计稿中某个入口位置或文案不稳定，以交互结构文档为准。

### 状态图 04：场景与待办详情

用于还原底部抽屉的暖白样式。重点：

- 抽屉从状态卡下方或屏幕底部浮出。
- 暖白底、深色文字、细分割线。
- 内容以记录块展示，不做深色卡片。

### 状态图 05：装备详情弹窗

用于还原装备弹窗。重点：

- 弹窗为暖白/浅灰底。
- 装备项为独立条目。
- 关闭、返回女骑详情逻辑清晰。

### 状态图 06：机动能力详情

用于还原指标型详情。重点：

- 体力、装备状态摘要、五项机动能力。
- 指标条保留高级仪表感。
- 不自行编造评级。

### 状态图 07 到 13

这些图作为详情层视觉参考。实现时允许根据真实变量和可读性优化布局，但必须保持：

- 暖白详情面。
- 低饱和蓝灰/冷蓝/暗红作为局部强调。
- 移动端阅读友好。
- 返回路径明确。
- 不出现伪功能。

## 图标方案

上一版纯手写 SVG 图标与设计稿差距较大。后续采用两层图标方案。

### 第一层：核心图标资产化

以下图标应优先生成透明背景 PNG/WebP 资产，而不是手写 SVG：

- 状态栏主盾徽 `crest`
- 主角盾徽 `protagonist`
- 骑士徽章 `knight`
- 女骑徽章 `female-rider`
- SAN 圆环或 SAN 核心图标 `san`
- 待办图标 `todo`
- 装备图标 `equipment`
- 机动能力图标 `mobility`
- 体力图标 `stamina`

资产要求：

- 风格接近设计稿：半拟物、徽章感、轻微体积、冷蓝/钢蓝/暗红强调。
- 透明背景。
- 至少提供 2 倍尺寸，避免移动端模糊。
- 命名放入状态栏资产目录，例如 `assets/icons/status/`。
- 不使用图标库截图。
- 不使用 emoji。

### 第二层：交互图标使用 SVG

以下简单交互图标继续使用 SVG：

- 展开
- 收起
- 返回
- 关闭
- 进入详情箭头

这些图标只承担控件功能，应保持简洁、清晰、可换色，不需要复杂体积。

## 现有代码可保留与应重做范围

### 可以保留

- `store.ts` 中 MVU 数据读取逻辑。
- Mock 预览模式。
- `OpenPanel` 和 `ExpandedCharacter` 的状态模型。
- 当前已梳理出的变量映射。
- 详情层的变量完整性。
- `状态栏交互结构文档.md` 中定义的交互状态。

### 应重做

- `App.vue` 的模板结构，至少需要围绕设计稿重新分区。
- `App.vue` 的样式系统。
- 当前深色人物卡列表。
- 当前横向 SAN 进度条。
- 当前线性 SVG 核心图标。
- 当前通用化的卡片样式。
- 当前过于工程化的标题与眉标。

## 视觉验收标准

实现后必须用 Mock 模式预览：

```text
dist/埃希斯特利亚/界面/状态栏/index.html?mock=1
```

验收步骤：

1. 打开 `状态图_01.png`。
2. 打开当前前端默认态截图。
3. 并排比较以下项目：
   - 深色主面板轮廓是否接近。
   - 顶部盾徽与标题是否接近。
   - 主角卡比例是否接近。
   - SAN 是否为圆环读数。
   - 场景三栏是否接近。
   - 待办卡是否为蓝色强调卡。
   - 底部人物区是否为暖白底板。
   - 骑士/女骑行是否接近设计稿。
4. 至少再抽查 `状态图_03.png` 和 `状态图_05.png`。
5. 若截图第一眼不像设计稿，不要进入细节修补，应该回到组件结构重做。

## 开发纪律

- 开发前先截图对照，不凭记忆写 UI。
- 每完成一个状态图对应状态，就截图比对。
- 不要因为现有代码已经能跑而迁就现有视觉。
- 不要把设计稿中的视觉问题照单全收，交互逻辑必须经过文档校正。
- 如果设计稿与交互结构文档冲突，先记录冲突，再选择更符合移动端阅读和变量边界的方案。
- 不要再使用前端通用图标库替代核心图标。
- 不要用纯线性 SVG 冒充设计稿里的核心徽章图标。
